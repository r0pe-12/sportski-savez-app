## 2. AI u dizajnu (arhitektura)

**Rizik:** over-engineering — AI ima tendenciju da predloži pattern-e koji "izgledaju profesionalno" (Repository pattern, sub-roles, multi-tenancy) iako nisu opravdani veličinom projekta.

**Konkretne odluke koje smo morali eksplicitno odbaciti (meta-plan §9):**

| AI predložio | Odluka | Razlog |
|---|---|---|
| Repository pattern preko Eloquent-a | NE | Eloquent već enkapsulira CRUD; dodatni sloj samo apstrakcija bez vrijednosti za projekat ove veličine |
| Sub-admin / per-school admin / read-only auditor | NE | Spec zahtijeva 3 role (Admin, Profesor, Student/Učenik) — širenje van toga je scope creep |
| Multi-tenancy škola | NE | Spec eksplicitno kaže "single-tenant, sve škole u istoj instanci" |
| Custom error catalog | NE | Laravel default error pages + Form Request validation poruke su dovoljne |
| Multi-language UI | NE | Samo `me` lokalna; Inertia `__()` helper je dovoljan |
| AZLP cleanup workflow | NE | Spec van obima — boolean polja postoje, workflow ne |

**Mitigacija:**
1. **NE-radi liste** u meta-planu §9 — eksplicitan registar šta NE pravimo, tako da AI ima jasnu granicu.
2. **YAGNI princip** u CLAUDE.md sekcija 8 — "Don't add features ... beyond what the task requires."
3. **Brainstorming sa korisnikom** prije large feature kreira "ne radi liste" za tu feature.

**Otvoreni rizik:** AI može da uvuče over-engineering u manjim mjerama (extra polje na modelu, extra method na servisu) koje promaknu reviewer-u jer nisu očigledne. Smanjeno preko Pest testova koji kažnjavaju nekorišten kod (`vendor/bin/pint --dirty` + dead-code detekcija).

## 3. AI u implementaciji (kod)

**Rizik:** code drift — AI ne čita postojeće konvencije ili pravi sitne devijacije (npr. `getUser()` umjesto `user()`, `User::admin()` umjesto `User::where('role', UserRole::Admin)`). Posebno opasno u SPA gdje frontend tipovi moraju da se slažu sa backend kontrolerima.

**Konkretni primjeri:**
- Sesija 18 (Implementation Phase 0/1) — AI je u prvom pokušaju kreirao migration koja je imala `email` kao `text` umjesto `varchar(255)` (default). Pint i Eloquent default su to popravili automatski; nijedan Pest test nije pao, ali bi u produkciji ovo bilo loše za indekse.
- Više sesija — AI je inicijalno koristio `findOrFail()` umjesto Laravel route-model binding-a. Konvencija je usaglašena pa popravljena.

**Mitigacija:**
1. **CLAUDE.md naming pravila** (sekcija 4) — tabele snake_case engleski, modeli PascalCase engleski, rute kebab-case, UI me kroz `__()`.
2. **Pint formatter** (`vendor/bin/pint --dirty --format agent`) je mandatory prije svake komite — uniformiše stil tako da AI ne može "vidjeti" tuđu konvenciju i drift-ovati.
3. **Wayfinder type safety** — `@laravel/vite-plugin-wayfinder` generiše TypeScript funkcije za sve Laravel rute. Frontend ne može da pozove nepostojeću rutu — TS compile pada.
4. **Pest testovi sa konvencija checks** — npr. `tests/Feature/RoutingTest.php` provjerava da svi admin URL-ovi imaju `role:admin` middleware.

## 4. AI u testiranju (Pest)

**Najopasniji rizik svih:** AI piše testove koji prolaze ali ne validiraju ništa stvarno. Test koji samo provjerava `assertTrue(true)` ili `expect($result)->not->toBeNull()` može da "preživi" review jer izgleda OK.

**Konkretne sumnjive situacije:**

1. **Tautološki test** — AI je u nekoliko slučajeva napisao test gdje su input i očekivani izlaz iste konstante. Primjer (pseudokod):
   ```php
   it('computes total', function () {
       $expected = 42;
       expect(compute(42))->toBe($expected); // nije test, kopija
   });
   ```
   Popravak: TDD red-green-refactor sa eksplicitnom verifikacijom da test prvo padne sa očekivanom porukom prije nego što se implementira logika.

2. **Test bez assertion-a** — Pest dozvoljava `it('does something', function () { doSomething(); });`. Prolazi ako exception nije bačen. AI je generisao nekoliko takvih u prvom draft-u Phase 2.

3. **Coupled test koji "prati" implementaciju** — kad AI generiše test koji testira `Service::compute()` tako što kopira logiku iz Service-a u test (umjesto da nezavisno izračuna očekivani rezultat). Test prolazi ali ne validira ništa.

**Mitigacija:**
1. **TDD red-green-refactor disciplina** — `superpowers:test-driven-development` skill nameće: prvo piši test koji pada, vidi failure poruku, pa implementiraj kod.
2. **Browser smoke testovi (Pest 4)** — `tests/Browser/SmokePagesTest.php` validira da stranice rade end-to-end, hvata regresije koje unit testovi ne mogu (npr. Inertia prop koji se zove drugačije u backend-u i frontend-u).
3. **Manual UAT za UC5 i UC8** — demo scenariji se ručno prolaze prije v1.0 tag-a, što hvata UX nedostatke koje testovi ne mogu (npr. confusing copy, missing helper text).

**Konkretan citat:** Sesija 18 je rebild-ovala 4 Pest testa za `TeamSubmitTest` jer su originalni testovi prolazili sa pogrešnim signature validation-om. Greška je otkrivena samo zato što je korisnik manuelno probao UC5 i vidio da kontroler prihvata "asdf" kao potpis. Test je popravljen, kontroler popravljen.

## 5. AI u deployment-u

**Rizik:** AI generiše `.env` vrijednosti, deployment skripte, ili infrastrukturne konfige koji "izgledaju" tačno ali nisu testirani na ciljnoj infrastrukturi. Posebno opasno za:
- `.env` produkcijske ključeve (može da unese `APP_KEY` koji se ne podudara sa serializovanim sesijama)
- Database migracije koje rade na SQLite a padaju na PostgreSQL (npr. `ENUM` umjesto `VARCHAR + CHECK`)
- Queue konfiguracije gdje `sync` driver radi u dev-u ali blokira request u produkciji

**Konkretni primjeri preventivnih mehanizama:**

1. **`.env.example` mora biti single source za env shape** — produkcijski `.env` se generiše iz template-a, ne kopira iz druge instance.
2. **Migracije moraju biti DB-agnostične** — koristi `Schema::table` API umjesto raw SQL. Spec piše schema agnostično (sekcija 7), implementacija prati.
3. **Database safety pravila** (CLAUDE.md sekcija 2.1) — zabranjene `migrate:fresh`, `db:wipe`, `migrate:refresh`, `migrate:rollback` ispod 2026_05_12_194833. Razlog: `ai_dnevnik_sesije` tabela čuva ADIS evidenciju.
4. **Pilot deployment** — staging environment (1 škola, 1 takmičenje) prije production rollout-a.

**Otvoreni rizik:** monitoring AI deployment-a u produkciji je ručan. Nemamo automatizovanu "deployment dry-run" koja bi uhvatila konfiguracijske greške prije production push-a.

**Šta dnevnik NE pokriva:**
- Subagent rad (samo glavni conversation upisuje, per CLAUDE.md sekcija 2.2)
- Promptove koji su odbačeni unutar iste sesije (samo finalni state se vidi)
- Real-time tool izlazi (samo "ishod" sumarum)

**Limitacija za regulator audit:** dnevnik je samostalan i ne sadrži cryptographic hash chain. Za regulatorni audit (AZLP, Ministarstvo prosvjete) bilo bi potrebno dodati `previous_hash` i append-only mehanizam. Trenutno je dovoljno za ADIS predaju (akademski kontekst).

## 7. Otvorena pitanja

Šta i dalje **ne znamo** o AI u SDLC nakon Phase 0–3:

1. **Long-context refaktoring pouzdanost** — AI sa 1M token context-om može da drži cijeli `app/` u kontekstu, ali ne znamo da li su izmjene koherentne kroz cijeli kod ili lokalno tačne ali globalno drift-ovane.
2. **Mjerenje AI bug rate-a** — nemamo automatizovan način da kažemo "X% bug-ova su AI greške naspram Y% ljudske greške". Sve baselineu nedostaje human-only kontrolna grupa.
3. **Da li je dnevnik dovoljan audit** — za akademsku predaju da, za regulatornu vjerovatno ne. Potrebno: append-only hash chain, tamper-evident storage.
4. **Skill konflikti** — kad više `superpowers:*` skill-ova istovremeno aktivnih, prioritet je nedeterministički. CLAUDE.md sekcija 9 mapira skill aktivaciju, ali konflikti se ručno rješavaju.
5. **AI cost vs human cost** — nemamo metriku "vrijeme uštede vs API trošak". Predmet ADIS izvještaja je AI **u** SDLC, ne ekonomska analiza.

## Troubleshooting

### "Unable to locate file in Vite manifest"

```bash
npm run build
```

### "Class 'Pdo_Sqlite' not found"

PHP nije kompajliran sa SQLite extenzijom. Ubuntu: `sudo apt-get install php8.3-sqlite3`. Windows: provjeri `php.ini` da je `extension=pdo_sqlite` uncommented.

### "Permission denied" za `storage/`

```bash
chmod -R 775 storage bootstrap/cache
```

### Wayfinder rute nedostaju u frontendu

```bash
npm run build
# ili tokom dev moda Wayfinder automatski regeneriše na change
```

### Queue worker neaktivan (UC5 OCR ne radi)

Provjeri da `php artisan queue:work` radi u zasebnom terminalu. Bez worker-a, `ValidateMedicalCertificateJob` ostaje u "pending" stanju i potvrda ne dobija status.

### Email se ne šalje

U dev modu email se piše u `storage/logs/laravel.log`. Provjeri tu, ne u inbox. Da se zaista šalje, podesi `MAIL_MAILER=smtp` u `.env` i konfiguri SMTP.

# Staging rollout — prijedlog (v1.0)

> **Pokriva profesorov zahtjev 3b:** prijedlog kako bi se sistem pustio u rad (pilot).

Cilj: pustiti aplikaciju na staging environment sa 1 školom i 1 takmičenjem prije produkcijskog rollout-a.

## Ciljna platforma: Laravel Cloud (Hobby tier)

**Zašto Laravel Cloud:**
- Best-in-class hosting za Laravel + Inertia (zero config)
- Autoscaling iz kutije
- Managed PostgreSQL + Redis + S3 + observability
- Vite build pipeline ugrađen

**Alternativa:** DigitalOcean App Platform, AWS Elastic Beanstalk, ili bare VPS sa Ploi/Forge. Cijena slična, više konfiguracije.

## Arhitektura staging-a

| Komponenta | Staging | Razlog |
|---|---|---|
| Host | Laravel Cloud Hobby ($10/mjesec) | Dovoljno za 1 školu i ~50 učenika |
| App tier | 1 instance, autoscaling off | Niska load |
| PHP | 8.3 FPM | Match dev |
| DB | PostgreSQL managed (1 GB) | Test migracija sa SQLite → PG |
| Cache/Queue | Redis (256 MB) | Match prod konfig, hvata sync vs queue greške |
| Storage | S3 bucket `skolski-sport-staging-storage` sa versioning on | Test private file storage |
| Email | AWS SES (sandbox) | Verifikovani recipient list ručno |
| OCR | FakeOcrAdapter (file-name konvencija) | Pravi Google Vision tek u produkciji |
| eDnevnik | FakeEDnevnikAdapter (deterministic) | Pravi adapter iza feature flag, ručno enable kad Ministarstvo da pristup |
| Monitoring | Laravel Pulse | Built-in, dovoljno za pilot |

## Koraci

### 1. Provision Laravel Cloud projekta

1. https://cloud.laravel.com/ → New Project
2. Connect GitHub repo `sportski-savez-app`
3. Region: `eu-central-1` (Frankfurt — najbliža CG)
4. Plan: Hobby

### 2. Konfiguriši environment varijable

U Laravel Cloud dashboardu → Environment → Add:

```env
APP_NAME="Sportski savez CG (Staging)"
APP_ENV=staging
APP_DEBUG=false
APP_URL=https://staging.skolski-sport-cg.app

LOG_CHANNEL=stack
LOG_STACK=daily

DB_CONNECTION=pgsql
# DB_* automatski popunjava Laravel Cloud kad attach-uješ DB

CACHE_DRIVER=redis
QUEUE_CONNECTION=redis
SESSION_DRIVER=redis

FILESYSTEM_DISK=s3
AWS_BUCKET=skolski-sport-staging-storage
# AWS_* iz Laravel Cloud secrets

MAIL_MAILER=ses
MAIL_FROM_ADDRESS=staging-notify@skolski-sport-cg.app

EDNEVNIK_ADAPTER=fake
OCR_ADAPTER=fake
```

### 3. Attach add-ons

U Laravel Cloud dashboardu:
- Database: PostgreSQL 16
- Cache: Redis 7
- Storage: S3 bucket (auto-create)

### 4. Deploy

```bash
git push origin main
```

Laravel Cloud automatski:
- `composer install --no-dev --optimize-autoloader`
- `npm ci && npm run build`
- `php artisan migrate --force`
- `php artisan db:seed --force` (samo pri prvom deploy-u, posle kapnuti)
- Start FPM + queue:work + scheduler

### 5. Health check

```bash
curl https://staging.skolski-sport-cg.app/up
```

Očekivano: 200 OK.

### 6. Pilot data setup

```bash
laravel-cloud ssh staging
php artisan tinker
```

```php
// Kreiraj 1 školu
$school = App\Models\School::factory()->create(['name' => 'Gimnazija Petar I', 'code' => 'GP1']);

// Kreiraj 1 admina za tu školu
App\Models\User::factory()->admin()->create(['email' => 'admin.gp1@savez.me']);

// Kreiraj 1 takmičenje
$sport = App\Models\Sport::factory()->create(['name' => 'Košarka', 'members_count' => 5, 'substitutes_count' => 3]);
App\Models\Competition::factory()->create(['sport_id' => $sport->id, 'name' => 'Pilot turnir 2026']);
```

### 7. UAT sa korisnicima

Pozvati 2-3 profesora iz škole pilot, daj kredencijale, traži da:
1. Registruje 5 učenika
2. Uploaduje ljekarske potvrde
3. Prijavi ekipu za pilot turnir
4. Verifikuje učenike kroz fake eDnevnik

Sakupi feedback i bug log u `docs/zavrsni-izvjestaj/deployment/uat-feedback.md` (van obima za predaju, planirano za narednu iteraciju).

## Rollback strategija

Ako staging deploy padne:

1. **DB migration rollback** — NE radimo (CLAUDE.md sekcija 2.1). Umjesto toga: kreiraj reverse migration koja vraća schema na prethodno stanje.
2. **App rollback** — `git revert` poslednjeg commit-a, push, Laravel Cloud redeploy.
3. **Backup restore** — Laravel Cloud auto-backup daily; ako se baza pokvari, restore iz dashboard-a.

# Production readiness checklist (v1.0)

> **Pokriva profesorov zahtjev 3b:** prijedlog kako bi se sistem pustio u rad (produkcija).

Cilj: prelaz iz staging-a u punu produkciju za sve škole u CG. Ne radi se u Phase 4 (van scope-a ADIS predaje), ovaj dokument je **prijedlog** za naručioca.

## Razlike staging vs production

| Tema | Staging | Production |
|---|---|---|
| Host | Laravel Cloud Hobby | Laravel Cloud Production tier (autoscaling on) |
| App tier | 1 instance | 2+ instances (min 2 za HA) |
| DB | PostgreSQL 1 GB | PostgreSQL 10+ GB + read replika |
| Cache/Queue | Redis 256 MB | Redis 1 GB+ + dedicated worker tier |
| Storage | S3 versioning on | S3 versioning + lifecycle 30 days + cross-region replication |
| Email | SES sandbox | SES production + verified domain |
| OCR | FakeOcrAdapter | Google Vision (production, rate limited 100 req/min) |
| eDnevnik | FakeAdapter | Pravi HTTP adapter, feature flag on |
| Monitoring | Pulse | Pulse + Sentry + CloudWatch + uptime monitoring (UptimeRobot) |
| Backup | Daily auto | Daily auto + weekly off-site (S3 cross-region) + monthly restore vježba |
| Pilot | 1 škola, 1 takmičenje | 2-3 škole, postepeni rollout, 3-mjesečni pilot |

## Pre-production checklist

### Pravni i administrativni

- [ ] **Sporazum sa Ministarstvom prosvjete za eDnevnik integraciju** — pisani sporazum o pristupu eDnevnik HTTP API-ju, scope (samo `getStudent(jmb)`), SLA i fallback ako API padne.
- [ ] **AZLP saglasnost framework** — usaglašena ToS + Privacy Policy + Data Processing Agreement sa AZLP (Agencija za zaštitu ličnih podataka).
- [ ] **Pravna validnost digitalnih potvrda** — sa Ministarstvom zdravlja: prihvatamo li skenirane ili samo elektronski potpisane PDF-ove? Trenutno: skenirane (OCR validacija).
- [ ] **Saglasnost roditelja workflow** — ako se uvodi (van Phase 4 scope-a), pravna analiza forme saglasnosti, retencija dokumenata.

### Tehnička pripravnost

- [ ] **Penetration test** — eksterni security audit, fokus na OWASP Top 10. Otkriveni nalazi popravljeni i retest pozitivan.
- [ ] **WCAG AA audit** — accessibility pregled svih stranica. Min: kontrast 4.5:1, keyboard navigacija, screen reader labels.
- [ ] **Load test** — `php artisan benchmark` ili k6 simulacija za peak load (300 ekipa po takmičenju, 100 paralelnih upload-a).
- [ ] **DR plan** — Disaster Recovery dokument: RPO 24h (data loss tolerance), RTO 4h (downtime tolerance). Restore procedura testirana.
- [ ] **Backup restore vježba** — weekly task: restore production backup na staging, validate integrity, log u `backup-restore-log.md`.
- [ ] **Smoke test posle deploy-a** — 5 ključnih URL-ova mora vratiti 200 OK posle svakog deploy-a:
  - `/` (home)
  - `/login`
  - `/dashboard` (kao admin)
  - `/teams` (kao profesor)
  - `/ai-dnevnik` (public)
- [ ] **Monitoring alarmi** — postavljen Sentry alert za error rate > 1%, CloudWatch alarm za DB CPU > 80%, uptime monitor za 5xx response > 5%.

### Operativna pripravnost

- [ ] **Runbook za incidente** — `docs/operations/runbook.md` (van scope-a Phase 4): šta uraditi kad DB padne, kad queue zaglavi, kad OCR rate-limit pogodi.
- [ ] **On-call rotacija** — 2+ admina obučena za debugging produkcije.
- [ ] **Korisnička podrška** — email alias `support@savez.me`, response SLA 24h tokom radnih dana.
- [ ] **Dokumentacija za korisnike** — video tutorijali ili PDF za 5 ključnih UC-ova (UC1, UC3, UC5, UC8, UC10). Snimci iz T4.3 mogu biti baza.

### Compliance

- [ ] **GDPR / AZLP saglasnosti** — checkbox u registraciji + privacy policy link + data export endpoint (`/profile/export` — van Phase 4 scope-a).
- [ ] **Data retention policy** — definisana retencija za audit_log (5 godina), medical_certificates (7 godina po AZLP), inactive accounts (3 godine).
- [ ] **Encryption at rest** — PostgreSQL TDE on, S3 SSE-S3 on, Redis AUTH on.
- [ ] **Encryption in transit** — HTTPS only, HSTS header, TLS 1.2 min.

## Rollout strategija

### Faza 1: 2-3 pilot škole (mjesec 1)

- Najveće 3 škole iz Podgorice (najveći volume, najlakši feedback loop)
- Daily standup sa pilot admin-ima prvih 2 nedjelje
- Sve UAT bug-ovi su P0/P1 prije širenja

### Faza 2: Region Podgorice + Nikšić (mjesec 2)

- Dodaj 5-10 škola
- Monitoring metrike: response time p95 < 500ms, error rate < 0.5%
- Korisnička podrška response SLA poštovan

### Faza 3: Cijela CG (mjesec 3+)

- Otključati registraciju za sve škole
- Marketing kroz Ministarstvo prosvjete (school admin onboarding webinar)
- Postavljen feature feedback kanal (Linear / GitHub Issues javno)

## Sunset / migration plan

Ako se sistem **menja** ili **gasi**:

1. **Notify korisnike 90 dana unaprijed** — email + in-app banner
2. **Export svih podataka** za škole — JSON arhiva sa svim njihovim učenicima, ekipama, rezultatima
3. **Sunset audit log** — finalni eksport `audit_log_entries` tabele, čuvati 5 godina po AZLP
4. **AI dnevnik** (`ai_dnevnik_sesije`) — academic artifact, čuvati indefinitely
